# Design — Bilingual (EN/ES) marketing site with next-intl

**Date:** 2026-05-12
**Status:** Approved (verbal). Ready for implementation planning.
**Scope:** Add a real Spanish version of the Xpandia marketing site alongside English, using `next-intl` (already a dependency, currently unused). Today the site is English-only and the header "EN | ES" toggle is cosmetic.

---

## Goals

1. Two locales: `en` (default) and `es`, on real, indexable URLs.
2. The header EN|ES toggle actually switches language (and stays on the same page).
3. All user-facing copy on the marketing pages, header, footer and FAB is translatable.
4. Spanish translations written now (professional draft; the client polishes later).
5. The blog (currently using a `?lang=` query param) is unified with the global locale system.
6. SEO done right: localized `<title>`/`<meta>`/OG per page, `hreflang` alternates, correct `<html lang>`.
7. English URLs and content do not change (no SEO regression).

## Non-goals

- No machine-translation pipeline or translation-management integration. Spanish copy lives in JSON files in the repo.
- No automatic locale detection / `Accept-Language` redirect — `/` is always English; switching is explicit. (Can be revisited later.)
- No per-locale OG image (one neutral logo OG image is fine for now).
- Not adding more locales beyond `en`/`es`.

---

## Decisions (from brainstorming)

| Question | Decision |
|---|---|
| Spanish copy source | Claude produces a professional draft now; externalize all strings; ship bilingual; client refines. |
| URL strategy | `localePrefix: 'as-needed'` — English no prefix (`/`, `/services`, …), Spanish `/es/…`. English URLs unchanged. |
| Blog | Unify with the global locale system: `/es/blog` ⇒ Spanish blog content (locale → `lang` passed to the Django backend). Remove the `?lang=` param and the standalone `BlogLanguageToggle`. |
| Message file layout | One JSON file per page/namespace under `messages/<locale>/` (Approach A). |
| Locale detection | Off (`localeDetection: false`). |
| Phasing | 3 phases with checkpoints (see below). |

---

## Architecture

### Routing

- `next.config.ts`: wrap the config with `createNextIntlPlugin('./i18n/request.ts')` (the repo has no `src/` dir, so config lives at `frontend/i18n/`).
- `i18n/routing.ts` — `defineRouting({ locales: ['en', 'es'], defaultLocale: 'en', localePrefix: 'as-needed' })`. The existing `lib/i18n/config.ts` (`SUPPORTED_LOCALES`, `DEFAULT_LOCALE`, `isValidLocale`, `LOCALE_LABELS`, `formatDate`) stays as the single source of truth for locale values; `i18n/routing.ts` imports from it (no duplication of the locale list).
- `i18n/navigation.ts` — re-export `Link`, `redirect`, `usePathname`, `useRouter`, `getPathname` from `createNavigation(routing)`.
- `i18n/request.ts` — `getRequestConfig` that, given the `requestLocale`, loads and shallow-merges `messages/<locale>/common.json`, `home.json`, `services.json`, `language-assurance.json`, `localization.json`, `aci.json`, `about.json`, `contact.json`, `blog.json` into one messages object (namespaced by file name).
- `middleware.ts` (repo root) — `createMiddleware(routing)` with `localeDetection: false`. Matcher excludes `/api`, `/_next`, `/media`, static files, and the special icon routes.
- `app/` → `app/[locale]/`:
  - `app/[locale]/layout.tsx` — was `app/layout.tsx`. Sets `<html lang={locale}>`, wraps children in `<NextIntlClientProvider messages={messages}>`. Calls `setRequestLocale(locale)` for static rendering. `generateStaticParams` returns `[{locale:'en'},{locale:'es'}]`. Validates the param (`notFound()` on unknown locale).
  - Move `app/page.tsx`, `app/providers.tsx`, `app/about/`, `app/contact/`, `app/services/`, `app/services/language-assurance/`, `app/services/localization-adaptation/`, `app/services/applied-cultural-intelligence/`, `app/blog/`, `app/blog/[slug]/` and all their `__tests__/` into `app/[locale]/…`.
  - `app/globals.css` stays at `app/globals.css`; import path in `app/[locale]/layout.tsx` becomes `../globals.css`.
  - Special files `app/icon.png`, `app/apple-icon.png` stay at `app/` (Next.js icon convention is root-level, locale-independent).
- Redirects in `next.config.ts`: keep `/services/qa|audit|fractional → /services/language-assurance` and add `/es/services/qa|audit|fractional → /es/services/language-assurance`.

### Messages / externalization

- `messages/en/*.json` and `messages/es/*.json`. `messages/es/*` initially = copies of `messages/en/*` (so the site works in Phase 1; `/es` shows English until Phase 2 fills them in).
- Files: `common.json` (header nav + dropdown labels, footer columns + tagline + description + copyright, FAB label, EN/ES toggle aria), `home.json`, `services.json`, `language-assurance.json`, `localization.json`, `aci.json`, `about.json`, `contact.json` (incl. all form field labels, radio options, microcopy), `blog.json` (hero copy per page, "Read more", pagination labels, empty state).
- Each page-namespace file also holds a `metadata` sub-object: `{ title, description, ogTitle, ogDescription }`.
- Components:
  - Server components use `getTranslations('home')` / `getMessages()`; client components (`contact/page.tsx`, `XpandiaHeader.tsx`) use `useTranslations('contact')`.
  - Structured arrays (the `engagements`, `coverageCards`, `criteriaCards`, etc. arrays of objects): the **shape** (numbers like `n: '01'`, `href`, image paths) stays in the component as a small typed array; only the **text fields** come from messages — either via `t('languageAssurance.engagements.0.title')` keyed by index, or `t.raw('languageAssurance.engagements')` returning a typed array of `{title, tagline, bestFor, …}`. Implementer picks whichever is cleaner per case; `t.raw()` is preferred for long lists.
  - `lib/i18n/config.ts#formatDate` etc. continue to be used for blog dates.
- All English copy currently in the components moves verbatim into `messages/en/*.json` (no copy changes — pure extraction). Spanish (Phase 2) is the translated equivalent.

### Locale switcher

- `XpandiaHeader.tsx`: the EN|ES pill group (desktop + mobile drawer) becomes a real switcher. On click, navigate to the current pathname under the other locale using `usePathname()` + `useRouter().replace(pathname, {locale})` from `i18n/navigation`. The active pill reflects `useLocale()`. Remove the `xpandia-lang` `localStorage` state and the related `useEffect` (the URL is now the source of truth).
- `<html lang>` is set by `app/[locale]/layout.tsx` from the route param.

### Blog (unified)

- `app/[locale]/blog/page.tsx` and `app/[locale]/blog/[slug]/page.tsx`: derive `lang` from the route `locale` (`useLocale()` / the `params.locale`), pass it to `fetchBlogPosts` / `fetchBlogPost`. Drop the `?lang=` `searchParams` handling.
- Remove `components/blog/BlogLanguageToggle.tsx` (and its test, its references) — the global header toggle covers it. `lib/services/blog.ts` keeps its `lang` parameter (still needed to ask the backend for localized content), it just gets fed from the locale now.
- Blog chrome strings (`HERO_COPY` etc.) move into `messages/<locale>/blog.json`. The "no posts" empty state copy is localized.
- Behavior when the backend has no Spanish post: same as today — show the localized empty/listing state. Out of scope: forcing English fallback content for missing Spanish posts.

### SEO / metadata

- Every page exports `generateMetadata({params})` that does `const t = await getTranslations({locale, namespace: '<page>.metadata'})` and returns `{ title, description, openGraph: {…}, alternates: { canonical, languages: { en: '…', es: '/es/…' } } }`. A small helper (`lib/seo/alternates.ts`) builds the `languages` map from the current path so it's not repeated.
- `app/[locale]/layout.tsx` `metadata` keeps `metadataBase` and the OG image; the title/description come from `common.json` or `home.json` metadata.
- `og-image.png` unchanged (neutral, logo-only).

### Tests / e2e

- A shared test util `frontend/test-utils/renderWithIntl.tsx` (or extend existing `jest.setup`) that wraps `render` with `NextIntlClientProvider` using the **English** messages (so existing assertions on English text keep working). `next-intl/navigation` mocked where needed (like the current `usePathname` mock).
- All moved page `__tests__/` and the layout tests adapted to render through the provider. New assertions: a smoke test that a page rendered with **Spanish** messages shows Spanish copy (once Phase 2 lands), and that the header switcher links to the `/es` path.
- E2E: new `e2e/public/i18n.spec.ts` — load `/`, click ES → URL becomes `/es`, H1 is Spanish, `document.documentElement.lang === 'es'`; click EN → back to `/`. Existing specs keep targeting the English (no-prefix) URLs unchanged. Add `LANGUAGE_SWITCH` / `i18n-locale-switch` to `flow-tags.ts` + `flow-definitions.json`. The blog specs adjust for the removed `?lang=`/toggle (use `/es/blog` instead).
- `lib/i18n/__tests__/config.test.ts` stays; add tests for `i18n/routing.ts` if it adds logic beyond config.

## Phasing (with checkpoints)

**Phase 1 — Infrastructure + English externalized.**
next-intl wired (`next.config`, `middleware.ts`, `i18n/routing.ts`, `i18n/navigation.ts`, `i18n/request.ts`); `app/` → `app/[locale]/`; `app/[locale]/layout.tsx` with provider + `<html lang>`; functional EN|ES switcher; all English copy moved to `messages/en/*.json`; `messages/es/*.json` = copies of English (fallback). Redirects updated. Unit tests adapted (rendered through the intl provider) and green. Dev server: `/`, `/es`, all routes 200; English content unchanged; `/es/...` shows English for now. → **checkpoint**.

**Phase 2 — Spanish translations.**
Fill `messages/es/*.json` with a professional Spanish translation, one namespace file at a time (parallel agents, like the PDF rewrite). Verify `/es/...` pages render Spanish copy. → **checkpoint**.

**Phase 3 — Blog unification + tests/e2e + final verification.**
Migrate blog to the locale routing; remove `?lang=` and `BlogLanguageToggle`; localize blog chrome; new `i18n.spec.ts`; update flow-tags/flow-definitions; update CLAUDE.md (routes, i18n note); full verification of EN and ES in the browser. → **checkpoint**.

---

## Risks / open items

- **Translation quality:** Xpandia is a Spanish language-quality firm; expect the client to revise the Spanish draft substantially. The draft should be high-quality, idiomatic LatAm-neutral Spanish, but it's explicitly a starting point.
- **Route move blast radius:** moving everything under `app/[locale]/` touches every page file's path; imports of `globals.css` and any relative imports shift by one level. Mitigated by doing it as Phase 1 with a checkpoint.
- **`generateStaticParams` / `setRequestLocale`:** needed for static rendering with next-intl 4; if any page uses `force-dynamic` it's fine, but the implementer must add `setRequestLocale(locale)` at the top of each `[locale]` page/layout per the next-intl docs.
- **Blog backend Spanish content:** depends on the Django backend actually having `es` posts. Not blocking — empty/listing state is acceptable.
- **`hello@xpandia.global`** and the `siteUrl` default (`https://xpandia.global`) are assumptions carried over from the PDF; confirm the real domain (affects `metadataBase`, `hreflang`, OG URLs).
